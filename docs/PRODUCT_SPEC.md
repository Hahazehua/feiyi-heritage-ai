# HeritageLink AI｜飞颐礼遇

## Wave 2 MVP 产品规格

## 1. 产品定位

HeritageLink AI 的长期目标是建立一个连接非遗商家与海内外礼品采购客户的 AI 匹配和定制服务平台。

平台帮助客户将模糊的文化礼赠需求转化为：

1. 可验证的结构化采购需求；
2. 符合已知商业约束的非遗产品推荐；
3. 有事实依据的中英文文化说明；
4. 商家可以直接确认、补充或报价的定制需求单。

本 MVP 重点验证：

* 用户需求能否被准确结构化；
* 用户信息不完整时，系统能否先提供有帮助的推荐；
* 用户明确提出的商业硬约束是否始终得到执行；
* 推荐结果是否合理、透明且可解释；
* 文化内容是否有来源且不编造事实；
* 定制需求单是否具有商家处理价值；
* API 不可用时核心演示是否仍可运行。

本产品采用“渐进式推荐”原则：

> 用户提供多少信息，系统就基于多少信息先给出结果；缺失信息不会阻止推荐，只会影响推荐模式、置信度和待确认事项。

---

## 2. 核心验证假设

### 2.1 用户侧假设

客户即使不了解具体非遗品类，也可以只描述部分需求，例如赠礼对象、场景或大致预算，先获得可理解的产品建议。

随着用户继续补充预算、数量、风格、交期和定制要求，推荐结果应当逐渐变得更准确。

### 2.2 商家侧假设

标准化的客户需求和定制需求单能够减少商家重复询问预算、数量、交期、定制内容和目的地等基础信息的成本。

即使用户没有一次性提供完整信息，系统也能够将已知信息和待确认事项清楚区分，方便商家继续沟通。

### 2.3 技术侧假设

自然语言模型适合负责：

* 理解用户需求；
* 累积多轮对话信息；
* 标记未知或模糊内容；
* 生成自然的补充问题。

产品可行性判断仍由确定性规则引擎负责。

### 2.4 平台侧假设

虽然首个试点只有飞颐铁画，但统一的数据结构、需求结构和推荐接口可以扩展到其他非遗商家和品类，而无需重写核心推荐流程。

---

## 3. 目标用户与场景优先级

### 3.1 首期用户

首期采购用户包括：

* 企业礼品采购人员；
* 政府、高校、博物馆等机构采购人员；
* 有商务答谢、纪念或文化礼赠需求的个人客户；
* 需要向海外对象赠送中国文化礼品的客户。

飞颐铁画工作人员在 MVP 中承担：

* 产品数据提供者；
* 产品事实审核者；
* 定制需求单接收者；
* 推荐结果和需求单可执行性的验证者。

### 3.2 优先演示场景

Wave 2 优先展示：

1. 企业面向海外合作伙伴的批量礼赠；
2. 商务答谢和周年纪念；
3. 个人向外国教授、朋友或长辈赠礼。

婚礼、乔迁、收藏等场景可以保留在产品选项中，但不作为首要路演案例。

### 3.3 MVP 不处理的海外问题

MVP 可以生成英文文化介绍并识别海外运输需求，但不对以下事项作正式承诺：

* 跨境支付；
* 关税和税务；
* 物流价格；
* 海关限制；
* 实际到货时间；
* 海外售后。

## 4. Wave 2 核心 Skill 与 Workflow

### Skill 1：自然语言礼品需求解析

将用户的一段中文或英文描述转换为结构化礼品采购需求。

输入示例：

> 我们想给 30 位美国合作伙伴准备周年纪念礼物，每件预算 1000 元，希望体现安徽文化，可以加入公司 Logo，30 天内完成，需要中英文介绍。

输出至少包括：

* 客户类型；
* 赠礼对象；
* 单件预算或总预算；
* 数量；
* 场景；
* 风格偏好；
* 文化寓意偏好；
* 定制要求；
* Logo 要求；
* 目的地；
* 海外运输要求；
* 交付周期；
* 输出语言；
* 已识别字段；
* 待补充字段；
* 不确定字段；
* 建议补充的问题；
* 当前推荐模式；
* 当前需求信息覆盖度。

模型只负责提取、合并和标准化需求，不得直接决定产品推荐结果。

### Skill 2：渐进式非遗产品推荐

系统根据当前已经确认或明确表达的需求信息：

1. 对用户明确提出的必要条件执行硬性过滤；
2. 对其他已知偏好进行可解释评分；
3. 根据当前信息完整度选择推荐模式；
4. 返回最多 3 个现有产品；
5. 展示每个维度的得分、状态和理由；
6. 标记哪些条件尚未提供或仍需商家确认；
7. 用户补充需求后自动更新推荐结果。

### Skill 3：对话式需求补充

系统在提供当前推荐的同时，可以提出一个最有价值的补充问题，例如：

> 这批礼品大约需要多少件？是否有单件预算范围？

补充问题的作用是优化推荐，而不是阻止推荐。

系统不得：

* 一次展示大量问题；
* 强迫用户填写全部字段；
* 重复询问已经回答的内容；
* 因为缺少非必要信息而拒绝展示任何产品。

### Skill 4：双语文化内容组织

根据产品数据库中已审核的事实和模板生成：

* 中文产品介绍；
* 英文产品介绍；
* 文化寓意；
* 制作工艺说明；
* 适合该客户场景的推荐表达。

当前版本不得加入数据库中不存在的产品事实。

### Skill 5：定制需求单生成

将用户需求与选中产品转换为结构化需求单，包括：

* 产品及价格快照；
* 数量；
* 目标预算；
* 定制主题；
* 题字内容；
* Logo 要求；
* 包装要求；
* 目的地；
* 交付期限；
* 待商家确认事项；
* 演示数据声明。

### 端到端 Workflow

```text
自然语言描述或详细表单
→ 结构化需求解析
→ 根据现有信息立即生成探索性或正式推荐
→ 可选追问一个高价值问题
→ 用户补充、修改或确认需求
→ 更新商业约束和推荐结果
→ 展示最多 3 个现有产品
→ 生成双语文化介绍
→ 无完全匹配产品时展示替代方案或定制概念
→ 生成并下载定制需求单
```

## 5. 核心用户流程

1. 用户进入 AI 礼品顾问。
2. 用户输入一段自然语言，或使用详细表单。
3. 系统提取并累积结构化需求。
4. 系统根据当前已知信息判断推荐模式。
5. 系统立即返回当前可用的产品建议，不要求用户先填写全部字段。
6. 如果补充某项信息能够明显提升推荐质量，系统同时提出一个可选的高价值问题。
7. 用户可以：

   * 直接查看当前推荐；
   * 回答补充问题；
   * 修改结构化需求；
   * 要求重新推荐；
   * 切换到详细表单。
8. 用户补充或修改信息后，系统重新执行过滤、评分和排序。
9. 系统返回最多 3 件现有产品。
10. 如果没有产品满足全部明确必要条件，系统展示：

    * 无完全匹配产品；
    * 冲突条件；
    * 接近条件的替代方案；
    * 可选的定制概念方案。
11. 用户选择产品或定制方向后，系统生成商家需求单。

## 6. 渐进式推荐模式

系统根据当前需求信息采用以下推荐模式。

### 6.1 探索推荐

适用于用户只提供少量信息，例如：

> 我想给外国朋友送一件有中国特色的礼物。

系统可以：

* 根据赠礼对象和文化场景展示代表性产品；
* 展示较低或中等推荐置信度；
* 标记预算、数量、交期等尚未提供；
* 提出一个可选补充问题；
* 不将结果表述为最终商业匹配。

### 6.2 引导推荐

适用于用户已经提供对象、场景，并提供部分预算、风格或文化诉求。

系统可以：

* 使用已知条件筛选和评分；
* 返回更有针对性的产品；
* 明确展示哪些维度参与了评分；
* 同时建议用户补充最有价值的信息。

### 6.3 约束推荐

适用于用户明确提出预算、数量、交期、Logo、定制或海外运输等必要条件。

系统必须：

* 严格执行用户明确提出的硬约束；
* 只把满足全部必要条件的产品列为完全匹配；
* 将存在冲突的产品放入独立的替代方案区域；
* 明确说明每项冲突，不得隐藏。

### 6.4 定制建议

当没有现有产品满足用户明确条件时，系统可以生成定制需求概念。

定制概念必须标记：

> 该内容为 AI 整理的定制需求概念，不代表现有产品、正式报价、产能或交付承诺。

---

## 7. 用户输入

所有字段均可由用户主动填写、在自然语言中表达，或在后续对话中补充。

### 7.1 商业与采购信息

* 预算类型：总预算或单件预算；
* 预算金额，币种默认为 CNY；
* 采购数量；
* 赠礼对象；
* 礼赠场景；
* 期望交付日期或交付天数；
* 目的地；
* 是否要求海外运输。

### 7.2 偏好与定制信息

* 客户类型；
* 风格偏好；
* 文化寓意偏好；
* 定制类型；
* 是否加入 Logo；
* 定制文字；
* 图案或主题方向；
* 包装要求；
* 输出语言；
* 其他说明。

枚举、字段类型和允许值以 `docs/DATA_SCHEMA.md` 为准。

### 7.3 自由文本处理原则

自由文本可以由需求解析 Skill 转换为结构化字段，但必须遵循：

* 明确表达的信息可以提取；
* 可以进行确定性换算，例如总预算除以数量；
* 未表达的信息不得自行补全；
* 模糊信息必须标记为不确定；
* 未提供的信息保持为空或标记为待补充；
* 缺失信息不得阻止探索性推荐；
* 用户明确表达的“必须、不能超过、需要、最晚”等条件必须作为硬约束；
* 原始用户文本必须保留，方便追溯；
* 用户修改后的结构化需求优先于模型初始解析结果。

## 8. 系统输出

### 8.1 结构化需求

包括：

* 原始用户文本；
* 标准化字段；
* 已识别字段；
* 待补充字段；
* 不确定字段；
* 可选补充问题；
* 当前推荐模式；
* 当前信息覆盖度；
* 用户最终修改版本。

### 8.2 推荐结果

系统输出 0–3 个完全匹配产品。

如果没有完全匹配产品，可以另外展示最多 3 个替代建议，但必须明确区分：

* 完全匹配；
* 当前推荐；
* 存在约束冲突的替代建议；
* 定制概念。

现有产品按以下规则排序：

1. `match_score` 降序；
2. 同分时按 `product_id` 升序。

每个结果必须包括：

* 产品基础信息；
* 推荐模式；
* 匹配分；
* 需求信息覆盖度；
* 八个维度的得分或状态；
* 命中标签；
* 推荐理由；
* 明确满足的条件；
* 未提供的条件；
* 限制条件；
* 风险和待确认项；
* 中文文化介绍；
* 英文文化介绍；
* 内容审核状态；
* 演示数据声明。

### 8.3 定制需求单

生成符合 `customization_inquiry` schema 的 JSON，必须包含：

* 用户需求快照；
* 产品或定制概念快照；
* 已明确的定制要求；
* 目的地及交期；
* 尚未提供的信息；
* 待商家确认问题；
* 已知事实与待确认信息的区分；
* MVP 演示声明。

## 9. 推荐方法

推荐采用以下三层结构：

```text
明确硬约束过滤
→ 基于已知偏好的可解释评分
→ 信息覆盖度和风险说明
```

### 9.1 明确硬约束过滤

只有用户明确表达为必要条件的信息才参与硬性过滤。

例如：

* “单件不能超过 1000 元”；
* “必须支持 Logo”；
* “最晚 30 天内交付”；
* “必须寄往美国”；
* “至少需要 50 件”。

必须过滤的情况包括：

* 产品最低演示单价超过用户明确的最高单件预算；
* 数量低于明确要求的采购数量或不满足起订量；
* 数量超过演示产能或允许数量；
* 用户明确要求的定制类型不受支持；
* Logo 为明确必要条件但产品不支持；
* 明确交期短于产品和定制所需制作周期；
* 海外运输为明确必要条件但产品未标记支持；
* 产品或关联商家数据无效；
* 产品未达到允许推荐的审核状态。

用户没有提供相关字段时：

* 不执行对应硬过滤；
* 不自行猜测；
* 产品可以进入候选集；
* 推荐结果必须显示该条件尚待确认。

### 9.2 可解释评分

推荐器使用以下基础权重：

| 维度      | 基础权重 |
| ------- | ---: |
| 预算匹配    |   25 |
| 赠礼对象匹配  |   15 |
| 场景匹配    |   15 |
| 风格匹配    |   15 |
| 文化寓意匹配  |   10 |
| 定制匹配    |   10 |
| 数量及产能余量 |    5 |
| 交付时间余量  |    5 |
| 合计      |  100 |

只有用户已经提供有效信息的维度参与个性化匹配。

未提供信息的维度标记为：

> 未提供，不参与本轮个性化评分。

为避免用户少填字段导致产品被错误判定为低匹配，系统应提供：

* `match_score`：基于当前有效维度归一化后的匹配分；
* `information_coverage`：本轮需求信息覆盖度；
* `confidence_level`：低、中、高。

不得把“信息未提供”直接视为“不匹配”。

### 9.3 探索模式排序

当用户提供的信息不足以形成个性化评分时，系统可以基于以下因素展示代表性产品：

* 产品数据有效；
* 内容审核状态；
* 场景适用范围；
* 产品信息完整度；
* 产品之间的主题和价格多样性。

探索模式必须明确说明：

> 当前结果主要用于帮助用户了解可选方向，并非基于完整采购条件的最终推荐。

### 9.4 替代建议

当产品违反用户明确提出的必要条件时，不得列入“完全匹配产品”。

系统可以在独立区域展示替代建议，但必须说明：

* 冲突了哪项条件；
* 需要用户调整什么；
* 是否可以通过商家定制解决。

## 10. AI 边界

当前 Wave 2 中：

* DeepSeek 负责自然语言需求理解；
* DeepSeek 负责合并多轮用户信息；
* DeepSeek 负责识别未知和不确定内容；
* DeepSeek 可以生成一个非强制性的补充问题；
* 规则引擎负责过滤、评分和排序；
* 模板模块负责双语内容组织；
* 产品数据库负责提供产品事实；
* 商家负责确认价格、产能、交期和定制可行性。

DeepSeek 不参与：

* 虚构或创建现有产品；
* 绕过明确硬约束；
* 修改规则评分；
* 决定产品真实价格；
* 决定真实产能；
* 决定真实交期；
* 决定真实运输能力；
* 补写未经验证的文化事实；
* 作出商家履约承诺。

AI 可以整理定制概念，但不得把定制概念伪装为现有产品。

## 11. 数据与事实来源

系统中的信息分为四类。

### A. 商家已确认事实

例如：

* 产品名称；
* 材料；
* 尺寸；
* 工艺；
* 定制能力；
* 经过确认的文化介绍。

可以用于推荐和内容展示。

### B. 有明确来源的公开文化资料

必须保存来源信息，并在进入正式版本前进行人工审核。

### C. MVP 演示数据

例如未经飞颐确认的价格、产能、交期或国际运输能力。

必须明确标记：

> MVP 演示数据

### D. AI 或模板生成表达

只能基于 A、B、C 类已有字段重新组织语言，不得新增产品事实。

内容必须带有：

* 来源状态；
* 审核状态；
* 待商家确认标记。

## 12. 对话与补充问题

### 12.1 补充问题原则

补充问题用于改善推荐，不用于阻止推荐。

系统应：

* 每轮最多提出一个高价值问题；
* 优先询问最能改变推荐结果的信息；
* 不重复询问用户已经回答的内容；
* 允许用户跳过问题；
* 用户跳过后仍展示当前推荐；
* 在页面上同步展示当前已知需求摘要。

### 12.2 建议询问顺序

在用户尚未提供对应信息时，建议优先询问：

1. 预算和数量；
2. 赠礼对象和使用场景；
3. 交付时间；
4. 定制和 Logo 要求；
5. 海外目的地及运输要求；
6. 风格和文化寓意；
7. 包装、题字和主题方向。

该顺序仅用于优化对话，不构成进入推荐的必要条件。

### 12.3 对话状态

## 13. 失败与回退处理

### 13.1 DeepSeek API 不可用

发生以下情况时：

* 未配置 API Key；
* API 超时；
* 认证失败；
* 余额不足；
* 网络错误；
* 空响应；
* 非法 JSON；
* 数据验证失败；

系统不得崩溃，应：

1. 显示非技术用户可理解的提示；
2. 明确标注当前使用演示解析模式；
3. 保留详细表单；
4. 允许用户继续获得探索性推荐；
5. 不展示 API Key、底层 Prompt、异常堆栈或模型内部推理。

### 13.2 无完全匹配产品

系统应返回：

* 当前没有满足全部明确条件的现有产品；
* 主要冲突条件；
* 可以调整的预算、数量、交期或定制要求；
* 独立展示的替代建议；
* 生成定制概念方案的入口。

不得为了展示结果而把冲突产品标记为完全匹配。

### 13.3 产品资料缺失

缺失事实显示为：

* 待商家确认；
* 暂无经审核信息。

不得生成替代性虚构内容。

## 16. Wave 2 验收指标

### 16.1 需求解析

* 明确表达的预算、数量和交期提取准确率目标不低于 90%；
* 定制、Logo、目的地和语言需求识别准确率目标不低于 90%；
* 未明确字段的自主编造率为 0；
* 所有未知或不确定字段均有标记；
* 用户提供的信息能够在多轮对话中正确累积。

### 16.2 渐进式推荐

* 用户仅提供部分信息时，系统仍能展示探索性或临时推荐；
* 缺失字段导致页面无结果的比例为 0；
* 未提供的信息不会被当作负面匹配；
* 当前推荐模式标记完整率为 100%；
* 信息覆盖度展示完整率为 100%；
* 用户补充信息后推荐结果能够正确刷新；
* 已回答字段重复询问率为 0；
* 用户跳过补充问题后仍可继续查看推荐。

### 16.3 数据质量

* 演示数据必填字段校验通过率 100%；
* ID 唯一性校验通过率 100%；
* 外键、枚举、价格范围和数据类型校验通过率 100%。

### 16.4 约束安全

* 用户明确提出的硬约束违规推荐数为 0；
* 冲突产品被错误标记为完全匹配的数量为 0；
* 无完全匹配产品时，替代建议均明确展示冲突条件；
* 相同输入和相同数据版本的排序一致率为 100%。

### 16.5 推荐质量

在至少 10 个经商家或领域人员标注的黄金需求中：

* Top-3 命中率目标不低于 80%；
* 每个推荐结果的匹配分和参与维度完整率为 100%；
* 每个推荐至少包含一个明确命中理由；
* 缺失信息不会被错误解释为用户偏好。

### 16.6 内容质量

* 推荐产品的中英文名称和文化介绍完整率为 100%；
* 内容来源或审核状态标记完整率为 100%；
* 未知产品事实编造率为 0。

### 16.7 定制需求单

* 所有导出文件通过 schema 校验；
* 已提供字段保存完整率为 100%；
* 未提供信息和待确认事项标记完整率为 100%；
* 已知事实与待确认事项区分率为 100%。

### 16.8 稳定性

* 没有 API Key 时应用仍可运行；
* DeepSeek 失败时详细表单和探索推荐仍可使用；
* 5–10 件产品规模下，非 API 推荐交互目标低于 1 秒；
* 自动化测试不得产生真实 API 费用；
* Streamlit 重运行后当前会话状态保持正常。

## 17. 非功能要求

* 默认中文界面；
* 产品文化内容同时提供英文；
* 代码字段使用英文 `snake_case`；
* 规则、数据、AI 客户端、对话管理、内容生成和 UI 解耦；
* 所有核心函数使用类型注解；
* 错误信息对非技术用户可理解；
* API Key 只从环境变量或部署平台 Secrets 读取；
* `.env`、私钥、缓存和敏感信息不得提交到仓库；
* 首屏、产品结果和导出文件均显示演示数据声明；
* 不收集完成推荐所不需要的个人信息；
* 用户可以跳过补充问题；
* 不得使用强制问答阻止用户查看产品。
