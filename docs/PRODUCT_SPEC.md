# HeritageLink AI｜飞颐礼遇

## Wave 2 MVP 产品规格

## 1. 产品定位

HeritageLink AI 的长期目标是建立一个连接非遗商家与海内外礼品采购客户的 AI 匹配和定制服务平台。

平台帮助客户将模糊的文化礼赠需求转化为：

1. 可验证的结构化采购需求；
2. 符合已知商业约束的非遗产品推荐；
3. 有事实依据的中英文文化说明；
4. 商家可以直接确认、补充或报价的定制需求单。

本 MVP 重点验证：

* 用户需求能否被准确结构化；
* 用户信息不完整时，系统能否先提供有帮助的推荐；
* 用户明确提出的商业硬约束是否始终得到执行；
* 推荐结果是否合理、透明且可解释；
* 文化内容是否有来源且不编造事实；
* 定制需求单是否具有商家处理价值；
* API 不可用时核心演示是否仍可运行。

本产品采用“渐进式推荐”原则：

> 用户提供多少信息，系统就基于多少信息先给出结果；缺失信息不会阻止推荐，只会影响推荐模式、置信度和待确认事项。

---

## 2. 核心验证假设

### 2.1 用户侧假设

客户即使不了解具体非遗品类，也可以只描述部分需求，例如赠礼对象、场景或大致预算，先获得可理解的产品建议。

随着用户继续补充预算、数量、风格、交期和定制要求，推荐结果应当逐渐变得更准确。

### 2.2 商家侧假设

标准化的客户需求和定制需求单能够减少商家重复询问预算、数量、交期、定制内容和目的地等基础信息的成本。

即使用户没有一次性提供完整信息，系统也能够将已知信息和待确认事项清楚区分，方便商家继续沟通。

### 2.3 技术侧假设

DeepSeek 在配置有效 API Key 时只作为可选的自然语言字段提取器，给出候选字段、未知项和不确定项。无 Key 或调用失败时，确定性演示解析器提供有限但可复现的回退。

多轮字段合并、类型与枚举校验、总预算换算、问题优先级、推荐就绪状态和需求签名均由本地 `dialogue_manager` 与 `request_parser` 决定；用户在确认页的修改拥有最高优先级。产品资格、过滤、评分和排序始终由确定性规则引擎负责。

### 2.4 平台侧假设

当前目录包含漆艺、龙泉青瓷、传统织绣和传统玉雕四个 `unverified` 演示分类，共 20 件带图 MVP 礼赠方案。目录由一个平台演示选品主体维护；统一的数据结构、需求结构和推荐接口仍保持主体与品类无关，但这不代表多商家后台或真实商家入驻已经实现。

## 3. 目标用户与场景优先级

### 3.1 首期用户

首期采购用户包括：

* 企业礼品采购人员；
* 政府、高校、博物馆等机构采购人员；
* 有商务答谢、纪念或文化礼赠需求的个人客户；
* 需要向海外对象赠送中国文化礼品的客户。

平台产品负责人及后续合作商家在 MVP 中承担：

* 产品数据提供者；
* 产品事实审核者；
* 定制需求单接收者；
* 推荐结果和需求单可执行性的验证者。

### 3.2 优先演示场景

Wave 2 优先展示：

1. 企业面向海外合作伙伴的批量礼赠；
2. 商务答谢和周年纪念；
3. 个人向外国教授、朋友或长辈赠礼。

婚礼、乔迁、收藏等场景可以保留在产品选项中，但不作为首要路演案例。

### 3.3 MVP 不处理的海外问题

MVP 可以生成英文文化介绍并识别海外运输需求，但不对以下事项作正式承诺：

* 跨境支付；
* 关税和税务；
* 物流价格；
* 海关限制；
* 实际到货时间；
* 海外售后。

## 4. Wave 2 Submitted Skills 与唯一 Workflow

本轮只提交以下四个 Skills。每个 Skill 的输入、输出、运行方式、代码、测试、回退和限制见 [`docs/wave2/skills/`](wave2/skills/)。

### Skill 1：Conversational Gift Request Understanding / 对话式礼赠需求理解

将首轮自然语言、确认页中的连续补充或详细表单转换为经过本地校验、可由用户修订确认的 `ParsedCustomerRequest`。

核心输出包括预算、数量、对象、场景、风格、文化寓意、定制、Logo、目的地、国际运输、交期、输出语言、已知字段、缺失字段、不确定字段和至多一个优先补充问题。DeepSeek 或确定性演示解析器只提供候选提取；`process_turn`、`merge_requests`、本地校验和用户确认维护权威累计状态。

详细说明：[`01-conversational-gift-request-understanding.md`](wave2/skills/01-conversational-gift-request-understanding.md)。

### Skill 2：Progressive Heritage Gift Recommendation / 渐进式非遗礼品推荐

只使用当前已知且有效的信息选择探索、引导或约束模式。系统对用户明确提出的必要条件执行硬过滤，对其他已知维度执行固定权重评分，展示信息覆盖度、置信度、八维状态和 0–3 件满足全部明确硬约束的目录内 MVP 演示方案。

如果没有方案满足全部明确硬约束，系统返回零件合格推荐，并把冲突方案放入独立参考区；不得为了填满结果而强行推荐。

详细说明：[`02-progressive-heritage-gift-recommendation.md`](wave2/skills/02-progressive-heritage-gift-recommendation.md)。

### Skill 3：Grounded Bilingual Heritage Content / 有事实边界的双语文化内容组织

从本地 CSV 中已有的中文、英文、来源说明和审核状态组织文化内容，不在运行时翻译或补写未知事实。当前 20 件方案对应 40 条双语资料，全部为 `review_status=draft`；页面必须标为 MVP 演示文案并提示待商家审核，不能称为商家已审核事实。

详细说明：[`03-grounded-bilingual-heritage-content.md`](wave2/skills/03-grounded-bilingual-heritage-content.md)。

### Skill 4：Merchant-Ready Customization Brief / 商家可执行的定制需求单生成

将用户已确认事实、选中的合格 MVP 演示方案快照、双语内容和待确认问题组装为可预览、复制和下载的 JSON 需求单。`InquiryRequestContext` 将未知预算、数量、定制、Logo、国际运输和交期保留为 `null` 或待确认项，不得用推荐内部代理值冒充客户事实。

无合格方案时生成的 `custom_heritage_gift_concept` 是独立的待商家评估概念，不等于已经选中产品的需求单，也不代表正式报价、产能或交付承诺。

详细说明：[`04-merchant-ready-customization-brief.md`](wave2/skills/04-merchant-ready-customization-brief.md)。

### Submitted Workflow

唯一 Submitted Workflow 为 [Conversational Heritage Gift Matching and Customization Workflow / 对话式非遗礼品匹配与定制工作流](wave2/WORKFLOW.md)：

```text
自然语言描述、连续补充或详细表单
→ 本地校验、累计与用户确认
→ 渐进式推荐和明确硬约束过滤
→ 0–3 件合格 MVP 演示方案，或明确的无完全匹配结果
→ 有事实边界的双语文化内容
→ 选中方案的商家可执行定制需求单
```

## 5. 核心用户流程

1. 用户进入 AI 礼品顾问。
2. 用户输入一段自然语言，或使用详细表单。
3. 系统在确认页显示连续对话记录；用户可以继续补充或修正，`process_turn` 在当前 session 中本地合并结构化需求。
4. 系统根据当前已知信息判断推荐模式。
5. 用户确认需求后，系统返回 0–3 件当前合格的 MVP 演示方案，不要求用户先填写全部字段。
6. 如果补充某项信息能够明显提升推荐质量，系统同时提出一个可选的高价值问题。
7. 用户可以：

   * 直接查看当前推荐；
   * 回答补充问题；
   * 修改结构化需求；
   * 要求重新推荐；
   * 切换到详细表单。
8. 用户补充或修改信息后，系统重新执行过滤、评分和排序。
9. 系统返回最多 3 件满足全部明确硬约束的目录方案。
10. 如果没有方案满足全部明确硬约束，系统展示：

    * 无完全匹配方案；
    * 冲突条件；
    * 接近条件的替代方案；
    * 可选的定制概念方案。
11. 用户选择一件合格目录方案后，系统生成商家需求单。无合格方案时可以生成独立定制概念，但该概念当前不进入“选中产品需求单”路径。

## 6. 渐进式推荐模式

系统根据当前需求信息采用以下推荐模式。

### 6.1 探索推荐

适用于用户只提供少量信息，例如：

> 我想给外国朋友送一件有中国特色的礼物。

系统可以：

* 根据赠礼对象和文化场景展示代表性产品；
* 展示较低或中等推荐置信度；
* 标记预算、数量、交期等尚未提供；
* 提出一个可选补充问题；
* 不将结果表述为最终商业匹配。

### 6.2 引导推荐

适用于用户已经提供对象、场景，并提供部分预算、风格或文化诉求。

系统可以：

* 使用已知条件筛选和评分；
* 返回更有针对性的产品；
* 明确展示哪些维度参与了评分；
* 同时建议用户补充最有价值的信息。

### 6.3 约束推荐

适用于用户明确提出预算、数量、交期、Logo、定制或海外运输等必要条件。

系统必须：

* 严格执行用户明确提出的硬约束；
* 只把满足全部明确必要条件的目录方案列为合格推荐；
* 将存在冲突的产品放入独立的替代方案区域；
* 明确说明每项冲突，不得隐藏。

### 6.4 定制建议

当没有目录方案满足用户全部明确硬约束时，系统可以生成独立的定制需求概念。

定制概念必须标记：

> 该内容为系统整理的定制需求概念，不代表现有产品、正式报价、产能或交付承诺。

## 7. 用户输入

所有字段均可由用户主动填写、在自然语言中表达，或在后续对话中补充。

### 7.1 商业与采购信息

* 预算类型：总预算或单件预算；
* 预算金额，币种默认为 CNY；
* 采购数量；
* 赠礼对象；
* 礼赠场景；
* 期望交付日期或交付天数；
* 目的地；
* 是否要求海外运输。

### 7.2 偏好与定制信息

* 客户类型；
* 风格偏好；
* 文化寓意偏好；
* 定制类型；
* 是否加入 Logo；
* 定制文字；
* 图案或主题方向；
* 包装要求；
* 输出语言；
* 其他说明。

枚举、字段类型和允许值以 `docs/DATA_SCHEMA.md` 为准。

### 7.3 自由文本处理原则

自由文本可以由需求解析 Skill 转换为结构化字段，但必须遵循：

* 明确表达的信息可以提取；
* 可以进行确定性换算，例如总预算除以数量；
* 未表达的信息不得自行补全；
* 模糊信息必须标记为不确定；
* 未提供的信息保持为空或标记为待补充；
* 缺失信息不得阻止探索性推荐；
* 用户明确表达的“必须、不能超过、需要、最晚”等条件必须作为硬约束；
* 原始用户文本必须保留，方便追溯；
* 用户修改后的结构化需求优先于模型初始解析结果。

## 8. 系统输出

### 8.1 结构化需求

包括：

* 原始用户文本；
* 标准化字段；
* 已识别字段；
* 待补充字段；
* 不确定字段；
* 可选补充问题；
* 当前推荐模式；
* 当前信息覆盖度；
* 用户最终修改版本。

### 8.2 推荐结果

系统输出 0–3 个满足全部明确硬约束的目录内 MVP 演示方案。“完全匹配”只表示没有违反本轮用户已明确的硬约束，不表示信息完整、真实在售、商家承诺或商业成功概率。

如果没有完全匹配方案，可以另外展示最多 3 个有明确冲突的参考建议，但必须明确区分：

* 满足全部明确硬约束；
* 当前推荐；
* 存在约束冲突的替代建议；
* 定制概念。

合格目录方案按以下规则排序：

1. `match_score` 降序；
2. 同分时按 `product_id` 升序。

每个结果必须包括：

* 产品基础信息；
* 推荐模式；
* 匹配分；
* 需求信息覆盖度；
* 八个维度的得分或状态；
* 命中标签；
* 推荐理由；
* 明确满足的条件；
* 未提供的条件；
* 限制条件；
* 风险和待确认项；
* 中文文化介绍；
* 英文文化介绍；
* 内容审核状态；
* 演示数据声明。

### 8.3 定制需求单

生成符合 `customization_inquiry` schema 的 JSON，必须包含：

* 用户需求快照；
* 一个选中目录方案的快照；
* 已明确的定制要求；
* 目的地及交期；
* 尚未提供的信息；
* 待商家确认问题；
* 已知事实与待确认信息的区分；
* 中英文 MVP 演示声明。

当前 `customization_inquiry` 必须且只能包含一个选中方案。无合格方案时生成的定制概念是另一种独立输出，不得伪装为选中产品需求单。

## 9. 推荐方法

推荐采用以下三层结构：

```text
明确硬约束过滤
→ 基于已知偏好的可解释评分
→ 信息覆盖度和风险说明
```

### 9.1 明确硬约束过滤

只有用户明确表达为必要条件的信息才参与硬性过滤。

例如：

* “单件不能超过 1000 元”；
* “必须支持 Logo”；
* “最晚 30 天内交付”；
* “必须寄往美国”；
* “至少需要 50 件”。

必须过滤的情况包括：

* 产品最低演示单价超过用户明确的最高单件预算；
* 用户采购数量低于方案最低起订量；
* 数量超过演示产能或允许数量；
* 用户明确要求的定制类型不受支持；
* Logo 为明确必要条件但产品不支持；
* 明确交期短于产品和定制所需制作周期；
* 海外运输为明确必要条件但产品未标记支持；
* 产品或关联商家数据无效；
* 产品未达到允许推荐的审核状态。

用户没有提供相关字段时：

* 不执行对应硬过滤；
* 不自行猜测；
* 产品可以进入候选集；
* 推荐结果必须显示该条件尚待确认。

### 9.2 可解释评分

推荐器使用以下基础权重：

| 维度      | 基础权重 |
| ------- | ---: |
| 预算匹配    |   25 |
| 赠礼对象匹配  |   15 |
| 场景匹配    |   15 |
| 风格匹配    |   15 |
| 文化寓意匹配  |   10 |
| 定制匹配    |   10 |
| 数量及产能余量 |    5 |
| 交付时间余量  |    5 |
| 合计      |  100 |

只有用户已经提供有效信息的维度参与个性化匹配。

未提供信息的维度标记为：

> 未提供，不参与本轮个性化评分。

为避免用户少填字段导致产品被错误判定为低匹配，系统应提供：

* `match_score`：基于当前有效维度归一化后的匹配分；
* `information_coverage`：本轮需求信息覆盖度；
* `confidence_level`：低、中、高。

不得把“信息未提供”直接视为“不匹配”。

### 9.3 探索模式排序

当没有任何维度参与个性化评分时，当前 Prototype 为每件合格方案使用中性展示分 50，再按 `product_id` 稳定排序。50 不代表中性命中、购买概率、成功率或业务指标；页面必须同时显示低信息覆盖度，且不得把未知维度标记为命中。

探索模式必须明确说明：

> 当前结果主要用于帮助用户了解可选方向，并非基于完整采购条件的最终推荐。

### 9.4 替代建议

当目录方案违反用户明确提出的必要条件时，不得列入“满足全部明确硬约束”的推荐区。

系统可以在独立区域展示替代建议，但必须说明：

* 冲突了哪项条件；
* 需要用户调整什么；
* 是否可以通过商家定制解决。

## 10. AI 边界

当前 Wave 2 中：

* DeepSeek 在配置有效 Key 时只提供自然语言字段提取候选、未知项和不确定项；
* 确定性演示解析器在无 Key 或 API 失败时提供有限、明确标记的本地回退；
* `dialogue_manager` 本地负责多轮字段合并、问题优先级、需求签名和推荐就绪状态；
* `request_parser` 本地负责字段白名单、类型、枚举、金额换算和交叉字段校验；
* 用户确认与修改是进入推荐前的权威需求版本；
* 规则引擎负责过滤、评分和排序；
* 模板模块负责双语内容组织；
* 本地 CSV 目录负责提供来源字段、MVP 演示方案字段和审核状态；
* 商家负责确认价格、产能、交期和定制可行性。

DeepSeek 不参与：

* 虚构或创建目录中的合格方案；
* 直接控制累计会话状态或推荐就绪状态；
* 绕过明确硬约束；
* 修改规则评分；
* 决定产品真实价格；
* 决定真实产能；
* 决定真实交期；
* 决定真实运输能力；
* 补写未经验证的文化事实；
* 作出商家履约承诺。

AI 表达可以用于整理定制概念，但概念对象由本地确定性代码构造，且不得把它伪装为现有产品、目录方案或选中产品需求单。

## 11. 数据与事实来源

系统中的信息分为四类。

### A. 商家已确认事实

例如：

* 产品名称；
* 材料；
* 尺寸；
* 工艺；
* 定制能力；
* 经过确认的文化介绍。

可以用于推荐和内容展示，但当前仓库没有可证明已由真实商家完成确认的 A 类商品记录；不得仅凭字段非空或 `status=active` 宣称商家已审核。

### B. 有明确来源的公开文化资料

当前 `data/catalog/heritage_products.csv` 保存开放馆藏来源页面、图片许可、历史元数据和 `museum_reference_not_for_sale` 状态。馆藏物件只作图片、工艺和文化资料参考，不是平台在售商品。公开资料必须保存来源信息，并在进入正式版本前进行人工审核。

### C. MVP 演示数据

当前 `data/demo/` 中的方案名称、价格、数量、交期、运输和定制能力均属于 MVP 演示数据，需要商家复核，不构成报价或履约承诺。

必须明确标记：

> MVP 演示数据

### D. AI 或模板生成表达

只能基于 A、B、C 类已有字段重新组织语言，不得新增产品事实。

内容必须带有：

* 来源状态；
* 审核状态；
* 待商家确认标记。

当前 20 件方案对应 40 条本地中英文文化资料，全部为 `review_status=draft`。运行时不调用 DeepSeek、机器翻译或 RAG 生成文化事实；页面必须把这些内容标为演示文案、待商家审核。

## 12. 对话与补充问题

### 12.1 补充问题原则

补充问题用于改善推荐，不用于阻止推荐。

系统应：

* 每轮最多提出一个高价值问题；
* 优先询问最能改变推荐结果的信息；
* 不重复询问用户已经回答的内容；
* 允许用户跳过问题；
* 用户跳过后仍展示当前推荐；
* 在页面上同步展示当前已知需求摘要。

### 12.2 建议询问顺序

在用户尚未提供对应信息时，建议优先询问：

1. 预算和数量；
2. 赠礼对象和使用场景；
3. 交付时间；
4. 定制和 Logo 要求；
5. 海外目的地及运输要求；
6. 风格和文化寓意；
7. 包装、题字和主题方向。

该顺序仅用于优化对话，不构成进入推荐的必要条件。

### 12.3 对话状态

当前 `ConversationState` 在一个 Streamlit session 内保存：

* `conversation_id`
* `messages`
* `raw_user_texts`
* `accumulated_request`
* `missing_blocking_fields`
* `missing_optional_fields`
* `uncertain_fields`
* `clarification_questions`
* `current_stage`
* `information_coverage`
* `last_recommendation_signature`
* `clarification_rounds`

首轮和确认页的后续补充均调用 `process_turn`；历史在确认页展开区用 Markdown 展示，补充输入通过显式“合并这条补充”操作更新累计需求。会话不写入正式数据库，也不提供账号、跨设备同步或长期历史记录。

## 13. 失败与回退处理

### 13.1 DeepSeek API 不可用

发生以下情况时：

* 未配置 API Key；
* API 超时；
* 认证失败；
* 余额不足；
* 网络错误；
* 空响应；
* 非法 JSON；
* 数据验证失败；

系统不得崩溃，应：

1. 显示非技术用户可理解的提示；
2. 明确标注当前使用演示解析模式；
3. 保留详细表单；
4. 允许用户继续获得探索性推荐；
5. 不展示 API Key、底层 Prompt、异常堆栈或模型内部推理。

### 13.2 无方案满足全部明确硬约束

系统应返回：

* 当前没有满足全部明确硬约束的目录方案；
* 主要冲突条件；
* 可以调整的预算、数量、交期或定制要求；
* 独立展示的替代建议；
* 生成定制概念方案的入口。

不得为了展示结果而把冲突方案标记为合格推荐。独立定制概念不包含虚构产品身份，也不进入选中产品需求单路径。

### 13.3 产品资料缺失

缺失事实显示为：

* 待商家确认；
* 暂无经审核信息。

不得生成替代性虚构内容。

## 16. Wave 2 验收口径与证据状态

以下条目是可由自动化测试或人工走查判定的验收断言，不是已经取得的百分比或排行榜成绩。实际命令、退出状态和人工观察统一记录在 [`docs/wave2/EVALUATION.md`](wave2/EVALUATION.md)，未执行前不得写成“已通过”。

### 16.1 Conversational Gift Request Understanding

* 明确表达的预算、数量、交期、定制、Logo、目的地和语言可以进入结构化字段；
* 未明确字段保持 `None`、空集合或待确认状态，不进行生成式补全；
* 总预算换算、类型、枚举和字段交叉关系由本地代码校验；
* 第二轮补充可以在当前 session 中合并到累计需求，用户修改优先；
* 无 API Key、超时、认证失败、空响应或非法 JSON 时安全回退到确定性演示解析。

### 16.2 Progressive Heritage Gift Recommendation

* 未知字段不启用对应硬过滤，也不作为不匹配；
* 用户明确的预算、数量、定制、Logo、交期和国际运输要求不可绕过；
* 返回结果不超过 3 件，并按当前匹配分降序、`product_id` 升序稳定排序；
* 页面展示推荐模式、信息覆盖度、低/中/高置信度和参与维度状态；
* 没有方案满足全部明确硬约束时返回零件合格推荐，并明确展示冲突和调整方向。

### 16.3 Grounded Bilingual Heritage Content

* 每个 active 目录方案必须有本地中文和英文记录；
* 内容模块只组织存储字段，不在运行时调用模型、机器翻译或 RAG 补写事实；
* 来源说明、`review_status` 和待确认字段随内容展示；
* 当前 40 条 `draft` 资料必须标为 MVP 演示文案、待商家审核。

### 16.4 Merchant-Ready Customization Brief

* 需求单只接受一个选中的合格目录方案，并保存选择时快照；
* `InquiryRequestContext` 只保存客户实际已知事实，未知预算、数量、定制、Logo、运输和交期保持 `null` 或进入待确认问题；
* 输出包含中英文 MVP 演示声明、双语内容、商家行动项和开放问题；
* JSON 在下载前通过 `validate_inquiry`，不得输出部分对象。

### 16.5 Workflow、数据与稳定性

* Streamlit 主路径可以从需求描述或详细表单到结构化确认、推荐、文化内容和需求单下载；
* 无满足条件的方案时不会出现选择合格方案的操作；
* CSV 加载校验必填列、主键、外键、枚举、金额、数量、双语行、图片路径和演示声明；
* 自动化测试使用占位 Key 和网络守卫，不调用真实外部 API；
* 没有 API Key 时仍可完成本地核心 Workflow。

### 16.6 当前未评测的业务指标

`tests/evaluation_cases.json` 当前是确定性回归案例集合，不包含商家或领域专家标注人、独立盲评或满意度证据。因此当前不报告自然语言解析准确率、Top-1/Top-3 命中率、商家满意度、转化率或履约成功率。未来若开展人工评测，必须先记录标注来源、计算方法和可复现结果，且不得把测试通过比例改写为业务指标。

## 17. 非功能要求

* 默认中文界面；
* 产品文化内容同时提供英文；
* 代码字段使用英文 `snake_case`；
* 规则、数据、AI 客户端、对话管理、内容生成和 UI 解耦；
* 所有核心函数使用类型注解；
* 错误信息对非技术用户可理解；
* API Key 只从环境变量或部署平台 Secrets 读取；
* `.env`、私钥、缓存和敏感信息不得提交到仓库；
* 首屏、产品结果和导出文件均显示演示数据声明；
* 不收集完成推荐所不需要的个人信息；
* 用户可以跳过补充问题；
* 不得使用强制问答阻止用户查看产品。
