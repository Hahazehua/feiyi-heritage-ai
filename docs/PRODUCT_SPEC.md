# HeritageLink AI｜非遗礼遇

## Wave 2 MVP 产品规格

## 1. 产品定位

HeritageLink AI 的长期目标是建立一个连接非遗商家与海内外礼品采购客户的 AI 匹配和定制服务平台。

平台帮助客户将模糊的文化礼赠需求转化为：

1. 可验证的结构化采购需求；
2. 符合商业约束的非遗产品推荐；
3. 有事实依据的中英文文化说明；
4. 商家可以直接确认、补充或报价的定制需求单。

本次 Wave 2 MVP 仅以飞颐铁画作为首个试点商家，使用 5–10 件明确标注为 `MVP 演示数据 / MVP demo data` 的产品，验证一条完整、可运行、可解释的交易前工作流。

本 MVP 不以成交规模、平台商家数量或复杂页面数量作为成功标准，而重点验证：

* 用户需求能否被准确结构化；
* 商业硬约束是否始终得到执行；
* 推荐结果是否合理且可解释；
* 文化内容是否有来源且不编造事实；
* 定制需求单是否具有商家处理价值；
* API 不可用时核心演示是否仍可运行。

---

## 2. 核心验证假设

本 MVP 需要验证以下假设：

### 用户侧假设

客户即使不了解具体非遗品类，也能够通过描述预算、对象、场景和文化诉求，获得可理解的产品建议。

### 商家侧假设

标准化的客户需求和定制需求单能够减少商家重复询问预算、数量、交期、定制内容和目的地等基础信息的成本。

### 技术侧假设

自然语言模型适合负责需求理解，但产品可行性判断必须由确定性规则引擎负责。

### 平台侧假设

虽然首个试点只有飞颐铁画，但统一的数据结构、需求结构和推荐接口可以扩展到其他非遗商家和品类，而无需重写核心推荐流程。

---

## 3. 目标用户与场景优先级

### 3.1 首期用户

首期采购用户包括：

* 企业礼品采购人员；
* 政府、高校、博物馆等机构采购人员；
* 有商务答谢、纪念或文化礼赠需求的个人客户；
* 需要向海外对象赠送中国文化礼品的客户。

飞颐铁画工作人员在 MVP 中承担：

* 产品数据提供者；
* 产品事实审核者；
* 定制需求单接收者；
* 推荐结果和需求单可执行性的验证者。

### 3.2 优先演示场景

Wave 2 优先展示：

1. 企业面向海外合作伙伴的批量礼赠；
2. 商务答谢和周年纪念；
3. 个人向外国教授、朋友或长辈赠礼。

婚礼、乔迁、收藏等场景可以保留在产品选项中，但不作为首要路演案例。

### 3.3 MVP 不处理的海外问题

MVP 可以生成英文文化介绍并识别海外运输需求，但不对以下事项作正式承诺：

* 跨境支付；
* 关税和税务；
* 物流价格；
* 海关限制；
* 实际到货时间；
* 海外售后。

## 4. Wave 2 核心 Skill 与 Workflow

### Skill 1：自然语言礼品需求解析

将用户的一段中文或英文描述转换为结构化礼品采购需求。

输入示例：

> 我们想给 30 位美国合作伙伴准备周年纪念礼物，每件预算 1000 元，希望体现安徽文化，可以加入公司 Logo，30 天内完成，需要中英文介绍。

输出至少包括：

* 客户类型；
* 赠礼对象；
* 单件预算或总预算；
* 数量；
* 场景；
* 风格偏好；
* 文化寓意偏好；
* 定制要求；
* Logo 要求；
* 目的地；
* 海外运输要求；
* 交付周期；
* 输出语言；
* 缺失字段；
* 不确定字段；
* 需要用户回答的问题。

模型只负责提取和标准化，不得决定推荐结果。

### Skill 2：非遗产品可行性过滤与推荐

根据结构化需求：

1. 执行商业硬约束过滤；
2. 对合格产品进行可解释评分；
3. 返回最多 3 个推荐产品；
4. 展示每个维度的得分和理由；
5. 无合格产品时解释冲突原因，不强行推荐。

### Skill 3：双语文化内容组织

根据产品数据库中已审核的事实和模板生成：

* 中文产品介绍；
* 英文产品介绍；
* 文化寓意；
* 制作工艺说明；
* 适合该客户场景的推荐表达。

当前版本不得加入数据库中不存在的产品事实。

### Skill 4：定制需求单生成

将用户需求与选中产品转换为结构化需求单，包括：

* 产品及价格快照；
* 数量；
* 目标预算；
* 定制主题；
* 题字内容；
* Logo 要求；
* 包装要求；
* 目的地；
* 交付期限；
* 待商家确认事项；
* 演示数据声明。

### 端到端 Workflow

```text
自然语言描述或详细表单
→ 结构化需求解析
→ 用户查看、修改和确认
→ 商业硬约束过滤
→ 可解释评分和排序
→ 展示最多 3 个产品
→ 生成双语文化介绍
→ 生成并下载定制需求单
```

## 5. 核心用户流程

1. 首页说明试点范围、演示数据性质及“不构成价格、库存或履约承诺”。
2. 用户选择输入模式：

   * 智能描述；
   * 详细表单。
3. 智能描述模式调用可配置的 DeepSeek API，将原文转换为结构化需求。
4. 没有 API Key、API 调用失败或用户选择演示模式时，系统使用确定性演示解析器。
5. 页面展示解析结果、缺失字段、不确定字段和补充问题。
6. 用户查看、修改并确认全部关键字段。
7. 系统校验预算、数量、场景和对象等必填项。
8. 总预算被换算成单件预算区间，但必须保留用户原始输入。
9. 规则引擎执行可行性过滤。
10. 对合格产品进行固定权重评分和稳定排序。
11. 页面展示最多 3 件产品及推荐解释。
12. 用户查看中英文文化内容。
13. 用户选择意向产品并生成结构化定制需求单。
14. 用户可在页面预览并下载 JSON 文件。
15. 无合格产品时，系统展示主要冲突和可调整建议。

## 6. 用户输入

### 6.1 必填字段

* 预算类型：总预算或单件预算；
* 预算金额，币种为 CNY；
* 采购数量；
* 赠礼对象；
* 礼赠场景。

### 6.2 选填字段

* 客户类型；
* 风格偏好；
* 文化寓意偏好；
* 定制类型；
* 是否必须加入 Logo；
* 是否必须支持海外运输；
* 定制文字；
* 图案或主题方向；
* 包装要求；
* 目的地；
* 期望交付日期或交付天数；
* 输出语言；
* 其他说明。

枚举、字段类型和允许值以 `docs/DATA_SCHEMA.md` 为准。

### 6.3 自由文本处理原则

自由文本可以由需求解析 Skill 转换为结构化字段，但必须遵循：

* 明确表达的信息可以提取；
* 未表达的信息不得自行补全；
* 模糊信息必须标记为不确定；
* 缺失信息必须标记为待用户确认；
* 用户确认前不得进入推荐引擎；
* 原始用户文本必须保留，方便追溯。

## 7. 系统输出

### 7.1 结构化需求

包括：

* 原始用户文本；
* 标准化字段；
* 解析模式；
* 缺失字段；
* 不确定字段；
* 补充问题；
* 用户最终确认版本。

### 7.2 推荐结果

输出 0–3 个产品，并按以下规则排序：

1. `total_score` 降序；
2. 同分时按 `product_id` 升序。

每个结果必须包括：

* 产品基础信息；
* 总分；
* 八个维度得分；
* 命中标签；
* 推荐理由；
* 限制条件；
* 风险和待确认项；
* 中文文化介绍；
* 英文文化介绍；
* 内容审核状态；
* 演示数据声明。

### 7.3 定制需求单

生成符合 `customization_inquiry` schema 的 JSON，必须包含：

* 用户需求快照；
* 产品及价格快照；
* 定制要求；
* 目的地及交期；
* 待商家确认问题；
* 已知事实与待确认信息的区分；
* MVP 演示声明。

---

## 8. 推荐方法

推荐采用“硬性过滤＋可解释规则评分”的两层结构。

### 8.1 硬性过滤

只有有效且满足必要条件的产品才能进入候选集。

必须过滤的情况包括：

* 产品最低演示单价超过用户绝对单件预算；
* 数量低于起订量；
* 数量超过演示产能或允许数量；
* 用户要求的定制类型不受支持；
* Logo 为必需条件但产品不支持；
* 必需交期短于产品和定制所需制作周期；
* 海外运输为必要条件但产品未标记支持；
* 产品或关联商家数据无效；
* 产品未达到允许推荐的审核状态。

缺失交期、目的地等信息时不得自行猜测，产品可以保留，但必须增加待确认风险。

### 8.2 可解释评分

通过硬性过滤的产品按照固定权重评分：

| 维度      |  权重 |
| ------- | --: |
| 预算匹配    |  25 |
| 赠礼对象匹配  |  15 |
| 场景匹配    |  15 |
| 风格匹配    |  15 |
| 文化寓意匹配  |  10 |
| 定制匹配    |  10 |
| 数量及产能余量 |   5 |
| 交付时间余量  |   5 |
| 合计      | 100 |

每个维度必须同时输出：

* 得分；
* 得分原因；
* 命中的产品标签；
* 仍需确认的风险。

### 8.3 AI 边界

当前 Wave 2 中：

* DeepSeek 负责需求字段提取；
* 规则引擎负责过滤、评分和排序；
* 模板模块负责双语内容组织；
* 产品数据库负责提供事实；
* 用户负责确认解析结果；
* 商家负责确认价格、产能、交期和定制可行性。

DeepSeek 不参与：

* 产品硬过滤；
* 最终评分；
* 最终排序；
* 产品价格判断；
* 产能判断；
* 交期判断；
* 文化事实补写；
* 商家履约承诺。

## 9. 数据与事实来源

系统中的信息分为四类：

### A. 商家已确认事实

例如：

* 产品名称；
* 材料；
* 尺寸；
* 工艺；
* 定制能力；
* 经过确认的文化介绍。

可以用于推荐和内容展示。

### B. 有明确来源的公开文化资料

必须保存来源信息，并在进入正式版本前进行人工审核。

### C. MVP 演示数据

例如未经飞颐确认的价格、产能、交期或国际运输能力。

必须明确标记：

> 仅用于 MVP 演示，不代表飞颐铁画的真实价格、产能、材料、交期、运输能力或商业承诺。

### D. AI 或模板生成表达

只能基于 A、B、C 类已有字段重新组织语言，不得新增产品事实。

内容必须带有：

* 来源状态；
* 审核状态；
* 待商家确认标记。

## 10. 失败与回退处理

### 10.1 DeepSeek API 不可用

发生以下情况时：

* 未配置 API Key；
* API 超时；
* 认证失败；
* 余额不足；
* 网络错误；
* 空响应；
* 非法 JSON；
* 数据验证失败；

系统不得崩溃，应：

1. 显示非技术用户可理解的提示；
2. 明确标注当前使用演示解析模式；
3. 保留详细表单；
4. 允许用户手动完成需求；
5. 不展示 API Key、底层 Prompt、异常堆栈或模型内部推理。

### 10.2 无合格产品

系统应返回：

* 无合格产品；
* 主要冲突条件；
* 可调整的预算、数量、交期或定制建议。

不得为了展示结果而绕过硬约束。

### 10.3 产品资料缺失

缺失事实显示为：

* 待商家确认；
* 暂无经审核信息。

不得生成替代性虚构内容。

## 13. Wave 2 验收指标

### 13.1 需求解析

* 明确表达的预算、数量和交期提取准确率目标不低于 90%；
* 定制、Logo、目的地和语言需求识别准确率目标不低于 90%；
* 未明确字段的自主编造率为 0；
* 所有缺失或不确定字段均有标记；
* 用户确认前调用推荐器的次数为 0。

### 13.2 数据质量

* 必填字段校验通过率 100%；
* ID 唯一性校验通过率 100%；
* 外键、枚举、价格范围和数据类型校验通过率 100%。

### 13.3 约束安全

* 自动测试中的硬约束违规推荐数为 0；
* 无合格产品时强行推荐率为 0；
* 相同输入和相同数据版本的排序一致率为 100%。

### 13.4 推荐质量

在至少 10 个经商家或领域人员标注的黄金需求中：

* Top-3 命中率目标不低于 80%；
* 每个推荐结果的总分和八维得分完整率为 100%；
* 每个推荐至少包含一个明确命中理由。

### 13.5 内容质量

* 推荐产品的中英文名称和文化介绍完整率为 100%；
* 内容来源或审核状态标记完整率为 100%；
* 未知产品事实编造率为 0。

### 13.6 定制需求单

* 所有导出文件通过 schema 校验；
* 必填字段完整率为 100%；
* 已知事实和待确认事项区分率为 100%。

### 13.7 稳定性

* 没有 API Key 时应用仍可运行；
* DeepSeek 失败时详细表单和推荐流程仍可使用；
* 5–10 件产品规模下，非 API 推荐交互目标低于 1 秒；
* 自动化测试不得产生真实 API 费用。

---

## 14. 非功能要求

* 默认中文界面；
* 产品文化内容同时提供英文；
* 代码字段使用英文 `snake_case`；
* 规则、数据、AI 客户端、内容生成和 UI 解耦；
* 所有核心函数使用类型注解；
* 错误信息对非技术用户可理解；
* API Key 只从环境变量或部署平台 Secrets 读取；
* `.env`、私钥、缓存和敏感信息不得提交到仓库；
* 首屏、产品结果和导出文件均显示演示数据声明；
* 不收集完成推荐所不需要的个人信息。
