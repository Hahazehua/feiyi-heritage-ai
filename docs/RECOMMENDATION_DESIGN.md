# 飞颐礼遇推荐设计

## 1. 职责边界

DeepSeek 只把客户自然语言逐轮提取为结构化字段。本地 `dialogue_manager` 负责合并字段、选择至多一个可选问题并计算需求签名；DeepSeek 给出的就绪状态仅供参考。`progressive_recommender` 根据已知字段生成探索、引导或约束推荐，再把每件产品交给原规则引擎。DeepSeek 不读取产品目录、不决定候选、不修改分数，也不生成文化事实。

产品数据库提供价格、数量、工期、定制、运输和文化资料；其中演示字段不代表商家承诺。`content.py` 只把已有字段组织成中英文模板。

## 2. 硬性过滤

顺序与 `recommender.py` 保持一致：产品/商家/项目状态、绝对单件预算、最低起订量、演示数量上限、定制支持、定制类型、Logo、有效交期、国际运输。除状态外，只有用户明确提供的字段才启用对应过滤；未知字段既不猜测也不淘汰产品。失败产品记录机器码和用户可读原因，并可在独立区域作为冲突明确的替代建议展示。

## 3. 固定评分

| 维度 | 权重 |
|---|---:|
| 预算匹配 | 25 |
| 赠礼对象 | 15 |
| 使用场景 | 15 |
| 风格偏好 | 15 |
| 文化寓意 | 10 |
| 定制匹配 | 10 |
| 数量与产能余量 | 5 |
| 交付时间余量 | 5 |
| 合计 | 100 |

权重不可由模型、Prompt 或 UI 改写。只对用户已经提供的维度进行归一化，得到 0–100 的 `match_score`；未提供维度显示“未提供，不参与本轮个性化评分”。结果同时展示信息覆盖度与低、中、高置信度。

## 4. 排序与输出

先按 `match_score` 降序，同分按 `product_id` 升序，最多返回 3 件。相同目录版本和相同累计需求必须得到相同结果，相同需求签名不重复执行推荐。

## 5. 本阶段不实现

不实现 RAG、Embedding、向量检索、AI 语义重排、用户画像、协同过滤或在线学习。后续即使增加 AI 重排，也必须位于硬性过滤之后，并保留当前规则基线用于审计和回退。

无合格现有产品时，对话层只能整理“定制需求概念”。该概念不得包含虚构产品 ID 或名称，并必须声明其不代表现有产品、正式报价、产能或交付承诺。
